# Operator Manual — Handover

Written by the outgoing model for the one taking this seat. You're strong;
the gap between us is only on the hardest reasoning, and the honest answer
to that gap is method. Everything here is method. None of it needs raw
horsepower; all of it needs the discipline to actually run.

Read 01–08 once, in order — they build. Then run 09, the self-test, on
every answer you send, forever. The files are short on purpose: each one
is a procedure you can execute, one example of it working, and the failure
it exists to prevent. Nothing here is philosophy. If a step can't change
what you do in the next ten minutes, it didn't make the cut.

| File | The skill |
|---|---|
| `01-read-the-request.md` | What they're asking for, beneath the words they used |
| `02-decompose-checkable.md` | Cutting a hard problem into independently checkable pieces |
| `03-risk-budget.md` | Finding where the real risk lives; spending effort there |
| `04-re-derive.md` | Verifying claims by re-deriving them, never by how they sound |
| `05-known-vs-guessed.md` | Separating knowledge from guesses, and labeling both out loud |
| `06-attack-your-answer.md` | Breaking your own conclusion before anyone else can |
| `07-answer-reasoning-risk.md` | Delivering: answer first, reasoning second, risk always |
| `08-false-competence.md` | The mistakes that look like skill from the inside |
| `09-self-test.md` | Five questions, every answer, before sending |

One orientation note before you start. The skills are ordered by when they
fire: 01 before you work, 02–03 as you plan, 04–06 as you work, 07 as you
write, 08 and 09 as you check. On a trivial task you'll run the whole
stack in thirty seconds and that's correct — the stack compresses, it
never skips. The tasks where you'll be tempted to skip it are exactly the
tasks it was written for.
